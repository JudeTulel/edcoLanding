Software Requirements Specification (SRS)

1. Introduction

1.1 Purpose

The purpose of this document is to define the requirements for the Rental Properties Finder and Management Application. This application will allow prospective tenants to find and rent apartments, while providing landlords with tools to manage their properties, view rent collections, and handle tenant requests.

1.2 Scope

The application consists of two main components:

  • Tenant Application: A mobile and web application for prospective tenants to search for rental properties, submit rental applications, and manage their rental agreements.
  • Landlord Application: A web application for landlords to list properties, manage tenant applications, track rent payments, and handle maintenance requests.

1.3 Definitions, Acronyms, and Abbreviations

  • Tenant: A person looking to rent or currently renting a property.
  • Landlord: The owner or manager of rental properties.
  • Property: A rental unit such as an apartment, house, or condominium.
  • SRS: Software Requirements Specification.

1.4 References

  • IEEE SRS Template
  • ISO/IEC/IEEE 29148:2011 Systems and software engineering

1.5 Overview

This SRS document includes an overall description, specific requirements, and appendices. It provides a detailed description of the functional and non-functional requirements for the Rental Properties Finder and Management Application.

2. Overall Description

2.1 Product Perspective

The application will be a standalone system accessible via web and mobile devices. It will interact with a central database to store and retrieve information about properties, tenants, and landlords.

2.2 Product Functions

Tenant Application:

  • Search and filter rental properties
  • View property details and photos
  • Submit rental applications
  • Sign rental agreements electronically
  • Make rent payments
  • Submit maintenance requests
  • Receive notifications

Landlord Application:

  • List and manage properties
  • View and approve/reject rental applications
  • Track rent payments and overdue payments
  • Handle maintenance requests
  • Generate financial reports
  • Communicate with tenants

2.3 User Classes and Characteristics

  • Tenants: Users looking to find and rent properties. They need an easy-to-use interface to search for properties and manage their rental agreements.
  • Landlords: Users managing one or multiple rental properties. They need tools to list properties, manage tenant applications, and handle financial transactions and maintenance requests.

2.4 Operating Environment

  • Tenant Application: Compatible with Android and iOS devices, and accessible via major web browsers (Chrome, Firefox, Safari).
  • Landlord Application: Accessible via major web browsers (Chrome, Firefox, Safari, Edge).

2.5 Design and Implementation Constraints

  • Compliance with data protection regulations (e.g., GDPR, CCPA)
  • Secure handling of payment information (PCI DSS compliance)
  • Responsive design to support various screen sizes and devices

2.6 User Documentation

  • User manuals and online help for both tenant and landlord applications
  • Video tutorials and FAQs

2.7 Assumptions and Dependencies

  • Reliable internet connection for accessing the applications
  • Users have basic proficiency with web and mobile applications

3. Specific Requirements

3.1 Functional Requirements

3.1.1 Tenant Application

  • Search Properties: The system shall allow tenants to search for properties based on various criteria (location, price range, number of bedrooms, etc.).
  • View Property Details: The system shall display detailed information about a selected property, including photos, description, rent amount, and amenities.
  • Submit Rental Application: The system shall enable tenants to submit a rental application for a property.
  • Sign Rental Agreements: The system shall allow tenants to sign rental agreements electronically.
  • Make Rent Payments: The system shall facilitate secure rent payments via multiple payment methods (credit card, bank transfer, etc.).
  • Submit Maintenance Requests: The system shall provide a form for tenants to submit maintenance requests.
  • Receive Notifications: The system shall send notifications to tenants for important events (application status, payment reminders, maintenance updates).

3.1.2 Landlord Application

  • List Properties: The system shall allow landlords to add and manage property listings.
  • Manage Applications: The system shall enable landlords to view, approve, or reject rental applications.
  • Track Rent Payments: The system shall track rent payments and alert landlords to overdue payments.
  • Handle Maintenance Requests: The system shall allow landlords to view and manage maintenance requests submitted by tenants.
  • Generate Reports: The system shall generate financial reports, including rent collection and expense summaries.
  • Communication: The system shall provide a messaging platform for landlords to communicate with tenants.

3.2 Non-Functional Requirements

3.2.1 Performance

  • The application should load within 3 seconds under normal conditions.
  • The system should handle up to 10,000 concurrent users.

3.2.2 Security

  • User data must be encrypted both in transit and at rest.
  • The application shall implement authentication and authorization mechanisms.
  • Regular security audits and vulnerability assessments shall be conducted.

3.2.3 Usability

  • The application shall have an intuitive user interface.
  • Accessibility features shall be implemented to support users with disabilities.

3.2.4 Reliability

  • The system shall have an uptime of 99.9%.
  • Data backup and recovery mechanisms shall be in place.

3.2.5 Scalability

  • The system shall be designed to scale horizontally to handle increased load.

3.3 Interface Requirements

3.3.1 User Interfaces

  • Tenant Application UI: Responsive design, easy navigation, search filters, property details, application form, payment interface, maintenance request form.
  • Landlord Application UI: Dashboard, property management, application review, payment tracking, maintenance management, reporting tools, messaging system.

3.3.2 Hardware Interfaces

  • No specific hardware interfaces are required.

3.3.3 Software Interfaces

  • Integration with payment gateways (e.g., Stripe, PayPal).
  • Integration with third-party services for electronic signatures (e.g., DocuSign).

3.3.4 Communication Interfaces

  • RESTful APIs for interaction between the client applications and the server.
  • WebSocket for real-time notifications.

4. Appendices

4.1 Glossary

  • API: Application Programming Interface
  • GDPR: General Data Protection Regulation
  • CCPA: California Consumer Privacy Act
  • PCI DSS: Payment Card Industry Data Security Standard

4.2 Analysis Models

4.3 Issues List

is EXPo better thn cli

Both Expo and the React Native CLI have their advantages and disadvantages, and the best choice depends on your specific project requirements and development preferences. Heres a comparison to help you decide which might be better for your rental properties finder and management application:

Expo

Advantages

  1. Ease of Setup and Use

    • Expo simplifies the setup process, making it easier to get started with a React Native project. It provides a managed workflow where a lot of the configuration is handled for you.

  2. Built-in Features

    • Expo comes with a lot of built-in APIs and components (e.g., camera, accelerometer, push notifications), which can save development time.

  3. Development Tools

    • Expo includes tools like Expo Go for quick testing on physical devices and Expo Snack for in-browser coding and sharing.

  4. OTA Updates

    • Expo supports over-the-air (OTA) updates, allowing you to push updates to your app without going through the app store approval process.

  5. Cross-platform Consistency

    • Expo ensures that your app runs consistently across both iOS and Android.

Disadvantages

  1. Limitations on Native Code

    • Expos managed workflow restricts you from adding custom native code or third-party libraries that require native modules. For advanced customizations, you might need to eject from Expo, which then requires more setup and maintenance.

  2. Larger App Size

    • Expo apps can be larger in size compared to those built with the React Native CLI because they include additional bundled libraries.

  3. Dependency on Expo SDK

    • Youre tied to the Expo SDKs release cycle, which may not always be up-to-date with the latest React Native features.

React Native CLI

Advantages

  1. Full Control

    • Using the React Native CLI gives you full control over your project, allowing you to integrate custom native modules and third-party libraries without restrictions.

  2. Customization

    • Greater flexibility for customization and optimization, which is beneficial for complex or performance-critical applications.

  3. Smaller App Size

    • Potentially smaller app sizes since you include only the libraries and modules your app needs.

Disadvantages

  1. Setup Complexity

    • Setting up a project with the React Native CLI can be more complex, requiring a proper environment setup (Xcode for iOS, Android Studio for Android).

  2. Manual Configuration

    • More manual configuration is required for linking native modules and handling different platform-specific requirements.

  3. Lack of Built-in Tools

    • Youll need to set up additional tools and libraries for things like OTA updates, testing, and development workflows that Expo provides out-of-the-box.

Which One to Choose?

Choose Expo if:

  • You are new to mobile development and want an easier setup and faster development process.
  • You need the convenience of built-in APIs and dont require extensive native code customization.
  • You want to take advantage of Expos OTA updates and development tools.
  • You are building a simple to moderately complex application.

Choose React Native CLI if:

  • You need full control over your project and plan to integrate custom native modules.
  • You are comfortable with a more complex setup process and manual configuration.
  • You are building a performance-critical or highly customized application.
  • You prefer to manage dependencies and project configuration yourself.

Conclusion

For a rental properties finder and management application, if you anticipate needing a lot of custom native features or third-party libraries that require native code, the React Native CLI might be the better choice. However, if you prefer a simpler setup with a lot of built-in features and tools, and you dont need extensive native customizations, Expo could be more suitable.

Ultimately, consider starting with Expo for its ease of use and seeing if it meets your requirements. If you hit any limitations, you can always eject to the React Native CLI later.
